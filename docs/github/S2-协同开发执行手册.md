# S2 协同开发执行手册（2026-04-21 至 2026-05-20）

## 1. 阶段定位

S2 是多格式输入和结构化抽取阶段。核心目标是把 PDF/Word 文档、图片页和截图统一接入同一条解析链路，OCR 必须进入主流程，最终输出可被 S3 直接消费的结构化字段与来源记录。

本阶段不追求规则结论，不追求完整 Web 体验。S2 的成功标准是：输入类型明确、来源可追溯、结构化字段可保存、解析结果可回放。

## 2. 入口条件

- S1 基线测试通过，接口总览已冻结 S2 入口字段。
- `StructuredField`、`ParseResultV2`、`OCRResultV2` 的语义已经确认。
- 至少准备 5 份样本报告，且样本里同时覆盖文本页和图片页。
- 本地环境脚本和 CI 可用。

## 3. 范围边界

必须完成：
- PDF 和 Word 的正文提取、页码映射和图片分离。
- OCR 单张和批量接口，且 OCR 结果进入主链路。
- OCR 后处理、字段清洗和来源合并。
- 结构化字段输出与来源记录保存。
- 解析、OCR 和字段回放回归测试。

明确不做：
- 不扩展复杂规则结论。
- 不做完整待复核工作流。
- 不做复杂 UI。
- 不做大规模性能优化。
- 不把 S2 输出改写成报告级最终结论。

## 4. 核心数据合同

S2 必须冻结以下字段语义：
- `field`：标准字段名，例如 `crypto.rsa.key_length`。
- `value`：可被规则比较的值，例如 `2048`。
- `source_type`：`text`、`image_ocr` 或 `word_text`。
- `page`：来源页码。
- `snippet`：来源片段。
- `confidence`：抽取置信度。
- `input_type`：输入来源类型，例如 `pdf`、`word`、`scan`。

字段输出示例：
```json
{
  "field": "crypto.tls.version",
  "value": "1.2",
  "source_type": "image_ocr",
  "page": 3,
  "snippet": "TLS1.2",
  "confidence": 0.91,
  "input_type": "pdf"
}
```

## 4.1 Word 解析实施约定

Word 文档解析在 S2 里必须按“结构化文本优先、版面信息补充”的方式实现：
- 优先提取正文段落、表格单元格和标题层级，不把 Word 当成普通纯文本文件直接读。
- 解析结果必须保留段落顺序，必要时保留页码或节信息，供后续字段回放使用。
- Word 中的嵌入图片必须分离出来，交给 OCR 主链路处理，不允许只抽正文不处理图片。
- Word 输入的 `input_type` 固定为 `word`，正文来源的 `source_type` 固定为 `word_text`。
- 同一份 Word 文档导出的 `StructuredField[]` 语义必须与 PDF 保持一致，不能因为输入格式不同而改变字段命名。

## 6. 联调主线

S2 主链路：
1. 上传或指定 PDF/Word。
2. parser 输出 `ParseResultV2`，其中 Word 文档需保留正文、表格与图片分离结果。
3. OCR 对图片输出 `OCRResultV2`。
4. 字段抽取器合并正文、Word 文本和 OCR，输出 `StructuredField[]`。
5. storage 保存 parse、OCR、field 记录。
6. web 查询任务时可看到解析状态、输入类型和字段数量。

联调检查点：
- 05-01：单 PDF/Word 文本和图片抽取稳定。
- 05-08：OCR 批处理和后处理初版可用。
- 05-12：结构化字段协议冻结。
- 05-15：5 份样本预验收。
- 05-20：M2 验收。

## 7. 质量门禁

每个 S2 PR 必须满足：
- parser/OCR 变更必须包含样本或构造测试。
- 字段名变更必须同步更新字段映射表。
- 存储接口必须保持调用方兼容。
- 批处理失败必须有错误记录，不允许静默丢失。
- 字段语义必须保持 `field/value/source_type/page/snippet/confidence/input_type` 一致。

M2 验收必须满足：
- S2 P0 Issue 全部关闭。
- 单报告解析与 OCR 链路完整运行。
- 5 份样本报告稳定通过。
- 输出字段包含字段名、值、来源类型、页码、片段、输入类型。
- CI 通过。

## 8. 风险控制

| 风险 | 处理方式 |
| --- | --- |
| 扫描件 OCR 不稳定 | 先保证原生 PDF、Word 和清晰截图，扫描件列为风险样本 |
| 图片分离失败 | 保留失败对象、页码和错误原因，便于人工定位 |
| 字段名混乱 | 字段映射表作为唯一标准，PR 中禁止临时字段名 |
| 批处理难调试 | 每份样本生成独立日志和中间结果 |

## 9. 退出标准

S2 完成后必须交给 S3 一个稳定输入：
- `StructuredField[]` 可直接进入规则引擎。
- 每个字段有来源页码、片段和输入类型。
- 存储层能回放一次解析结果。
- OCR 低置信度项能标记出来，供 S3 待复核使用。
