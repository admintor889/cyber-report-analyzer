# S2 协同开发执行手册（2026-04-21 至 2026-05-20）

## 1. 阶段定位

S2 是数据链路建设阶段。核心目标是把“PDF 输入”变成可被规则引擎消费的结构化字段，并保留字段来源，为 S3 的条款级判定提供稳定输入。

本阶段不追求规则覆盖率，也不追求 Web 完整体验。S2 的成功标准是：数据能稳定提取、能定位来源、能保存和回放。

## 2. 入口条件

- S1 的 parser、model、rules、web 基线测试通过。
- `StructuredField` 字段语义已在接口文档中确认。
- 至少准备 5 份样本报告或构造样本。
- 本地环境脚本和 CI 可用。

## 3. 范围边界

必须完成：
- PDF 正文提取和图片分离稳定化。
- OCR 单张和批量接口。
- OCR 后处理和字段清洗。
- 结构化字段输出。
- 结构化记录保存。
- 解析和 OCR 回归测试。

明确不做：
- 不扩展复杂规则结论。
- 不做完整待复核工作流。
- 不做复杂 UI。
- 不做大规模性能优化。
- 不把 S2 输出改写成报告级最终结论。

## 4. 核心数据合同

S2 必须冻结以下字段语义：
- `field`：标准字段名，例如 `crypto.rsa.key_length`。
- `value`：可被规则比较的值，例如 `2048`。
- `source_type`：`text` 或 `image_ocr`。
- `page`：来源页码。
- `snippet`：来源片段。
- `confidence`：抽取置信度。

字段输出示例：
```json
{
  "field": "crypto.tls.version",
  "value": "1.2",
  "source_type": "image_ocr",
  "page": 3,
  "snippet": "TLS1.2",
  "confidence": 0.91
}
```

## 6. 联调主线

S2 主链路：
1. 上传或指定 PDF。
2. parser 输出 `ParseResultV2`。
3. OCR 对图片输出 `OCRResultV2`。
4. 字段抽取器合并正文和 OCR，输出 `StructuredField[]`。
5. storage 保存 parse、OCR、field 记录。
6. web 查询任务时可看到解析状态和字段数量。

联调检查点：
- 05-01：单 PDF 文本和图片抽取稳定。
- 05-08：OCR 批处理和后处理初版可用。
- 05-12：结构化字段协议冻结。
- 05-15：5 份样本预验收。
- 05-20：M2 验收。

## 7. 质量门禁

每个 S2 PR 必须满足：
- parser/OCR 变更必须包含样本或构造测试。
- 字段名变更必须同步更新字段映射表。
- 存储接口必须保持 S1 调用方兼容。
- 批处理失败必须有错误记录，不允许静默丢失。
- 字段语义必须保持 `field/value/source_type/page/snippet/confidence` 一致。

M2 验收必须满足：
- S2 P0 Issue 全部关闭。
- 单报告解析与 OCR 链路完整运行。
- 5 份样本报告稳定通过。
- 输出字段包含字段名、值、来源类型、页码、片段。
- CI 通过。

## 8. 风险控制

| 风险 | 处理方式 |
| --- | --- |
| 扫描件 OCR 不稳定 | 先保证原生 PDF 和清晰截图，扫描件列为风险样本 |
| 图片分离失败 | 保留失败对象、页码和错误原因，便于人工定位 |
| 字段名混乱 | 字段映射表作为唯一标准，PR 中禁止临时字段名 |
| 批处理难调试 | 每份样本生成独立日志和中间结果 |

## 9. 退出标准

S2 完成后必须交给 S3 一个稳定输入：
- `StructuredField[]` 可直接进入规则引擎。
- 每个字段有来源页码和片段。
- 存储层能回放一次解析结果。
- OCR 低置信度项能标记出来，供 S3 待复核使用。
