# 阶段协同文档导航

## 总体原则

本项目按“先稳定数据链路，再扩大规则能力，最后完成可演示系统”的顺序推进。

工程原则：
- 规则优先：硬性安全结论优先由规则引擎给出，大模型只做归一化、解释和待复核辅助。
- 证据优先：任何 `FAIL` 或 `REVIEW` 结论都必须能追溯到页码、文本片段或截图来源。
- 小步集成：每个 PR 只解决 1 到 3 个逻辑点，必须附测试证据。
- 接口先行：每阶段开始前冻结本阶段 A 级字段，阶段内只增不删。
- 验收可复现：每个里程碑都必须能用命令、样本或演示脚本复现。

## 质量门禁

每个阶段的合并门禁：
- `python -m compileall src tests scripts` 必须通过。
- `pytest -q` 必须通过。
- 涉及接口字段变更时，必须同步更新 `docs/architecture/stage-interfaces/`。
- 涉及规则变更时，必须同步更新 `docs/rules/` 和 `tests/rules/`。
- 涉及解析或 OCR 变更时，必须附最小样本输入和预期输出。

每个阶段的验收门禁：
- P0 Issue 必须全部关闭。
- P1 Issue 完成率必须达到本阶段验收标准。
- 演示脚本或复现命令必须可运行。
- 阶段总结必须列出已完成项、遗留风险、下阶段依赖。

## 依赖关系

- S2 的字段结构直接决定 S3 规则输入，S2 不允许随意改变 `field/value/source/page/snippet/confidence` 语义。
- S3 的规则结果和证据结构直接决定 S4 页面展示，S3 结束前必须冻结 `RuleResult` 和 `EvidenceTrace`。
- S4 的导出内容必须与页面展示一致，否则 S5 不能进入发布冻结。
- S5 不再接受高风险新功能，只允许修复阻断缺陷、补文档和修演示脚本。

## 预验收节奏

- 每个阶段结束前 5 天完成预验收。
- 每个阶段结束前 3 天冻结新增功能。
- 每个阶段结束前 2 天只允许修复阻断问题和文档不一致。
- 里程碑当天只做验收、打标签、归档，不做大规模开发。

## 建议使用顺序

1. 先读本文档，确认阶段目标、依赖关系和质量门禁。
2. 再读本阶段执行手册，确认入口条件、范围边界、交付物和退出标准。
3. 根据本阶段 Issue 清单批量建任务并打标签。
4. 每周按固定节奏推进并更新 Project 看板。
5. 阶段末按里程碑清单完成预验收与正式验收。
