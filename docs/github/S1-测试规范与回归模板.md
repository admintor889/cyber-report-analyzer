# S1 测试规范与回归模板

## 1. 目标

本文件用于约束 S1 阶段的最小测试与回归流程，确保 parser、model、rules、web、reporting 等基础模块在协同提交后仍可运行、可验证、可交接。

S1 测试目标不是覆盖完整产品能力，而是保障工程基线稳定：
- 代码能通过语法检查。
- 单元测试和最小端到端测试可重复执行。
- PR 关闭前有清晰的测试证据。
- 缺陷发现后能记录、复现、回归。

## 2. 测试范围

S1 必测范围：
- parser：PDF 文本抽取和图片分离 MVP。
- model_review：RSA、TLS、弱算法关键词归一化。
- rules_engine：S1 首版规则台账中的 P0/P1 规则边界。
- web：任务创建、任务查询、结果回填最小流程。
- reporting：三态结果统计和摘要导出。
- e2e：parser、model、rules、web、reporting 的最小链路冒烟。

S1 暂不强制测试范围：
- 全量 OCR 识别准确率。
- 数据库存储和权限控制。
- 大规模并发和性能压测。
- 完整 Web 页面交互。

## 3. 本地测试命令

开发者提交 PR 前必须在仓库根目录执行：

```powershell
python -m compileall src tests
python -m pytest -q
```

如果只改文档，可以不跑完整测试，但 PR 或 Issue 关闭说明中必须写明：
- 本次只改文档。
- 未运行测试的原因。
- 受影响的文档路径。

## 4. CI 要求

S1 CI 至少包含以下步骤：
- checkout 仓库代码。
- 安装 `requirements.txt` 中的依赖。
- 执行 `python -m compileall src tests`。
- 执行 `pytest -q`。

CI 触发要求：
- 面向 `main` 或 `develop` 的 PR 必须触发。
- CI 失败时不得合并，除非负责人确认失败与本次变更无关，并记录原因。

## 7. 回归流程

缺陷修复后必须执行以下回归流程：
1. 用原始复现步骤确认缺陷已修复。
2. 执行受影响模块测试。
3. 执行 `python -m pytest -q`。
4. 在 Issue 或 PR 中补充回归结果。
5. 如果修复影响接口或字段，必须同步更新对应文档。

## 8. S1 验收口径

S1 测试验收通过条件：
- `.github/workflows/ci.yml` 存在并能执行语法检查与测试。
- `tests/` 下具备 parser、model_review、rules_engine、web、e2e 等基础测试。
- 本地 `pytest -q` 可通过。
- 失败或跳过的测试有明确说明。
- 缺陷记录和回归模板可直接用于 Issue/PR 关闭。

如果存在跳过测试，需要在验收时说明跳过原因。示例：
- 缺少真实 OCR 样例图，实图 OCR 测试跳过。
- 外部依赖未纳入 S1 范围，相关测试延后到 S2。
