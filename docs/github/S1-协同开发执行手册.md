# S1 协同开发执行手册（2026-03-31 至 2026-04-20）

## 1. 阶段定位

S1 是项目启动和工程基线阶段，目标不是做完整产品，而是把需求边界、模块接口、规则边界、测试基线和最小可运行链路建立起来。

本阶段结束后，团队必须能回答：
- 这个工具要解决什么问题，不解决什么问题。
- PDF、规则、模型、后端之间如何传数据。
- 首版必须支持哪些高优先级规则。
- 后续 S2/S3/S4 分别依赖哪些接口和字段。
- 标准表、检查项与条款级结果如何映射。

## 2. 入口条件

- 项目材料已归档，包括申报表、任务书、企业 checklist、计划书。
- 仓库目录结构已建立。
- GitHub Issue、PR、CI 基线可用。
- 团队成员已按模块认领职责。

## 3. 范围边界

必须完成：
- 需求边界冻结。
- 首版规则台账 v0.1。
- 模块接口和阶段接口文档。
- PDF 文本和图片抽取 MVP。
- 模型归一化最小规则。
- 标准表、检查项和结果映射规范。
- 后端任务接口骨架。
- UI / UE 原型落地和基础权限管理。
- CI、测试和本地环境脚本。

明确不做：
- 不做 report-level 一票否决机制。
- 不做 deviation / Not Applicable 自动推导。
- 不做全量 OCR 效果优化。
- 不做全量规则自动化。
- 不做企业级多租户和复杂权限。
- 不做企业级部署。

## 5. 联调主线

S1 最小联调链路：
1. `parser.extract_text_and_images(Path)` 从 PDF 输出 `text_blocks` 和 `image_paths`。
2. `rules_engine.s1_field_extractor.extract_s1_fields_from_text()` 抽取条款级字段。
3. `model_review.semantic_review(dict)` 对 RSA、TLS、弱算法文本做归一化。
4. `rules_engine.evaluate_s1_baseline(dict)` 输出条款级 `PASS/FAIL/REVIEW`。
5. `web.submit_report()` 创建任务，`web.set_task_result()` 回填结果。
6. `reporting.export_summary()` 统计三态结果。

联调输出必须包含：
- `task_id`
- `fields`
- `rule_results`
- `summary`
- `review_details` 或待复核信息
- `findings` 或条款级不符合项列表

## 6. 质量门禁

每个 S1 PR 必须满足：
- 有关联 Issue。
- 只改本阶段范围内的文件。
- 新增或修改代码必须有测试。
- `pytest -q` 通过。
- 条款级检查口径与标准表一致。
- 涉及接口或字段变更时同步更新文档。
- 不得将单条 `FAIL` 解释为整份报告失败。

S1 里程碑验收必须满足：
- P0 Issue 全部关闭。
- P1 Issue 完成率不低于 80%。
- CI 可运行并通过。
- 文档、代码、测试三者一致。
- 可以演示标准表、检查项、结果映射和条款级输出。
- 可以演示模型归一化、后端任务流程和 parser MVP。
- 检查正确率达到 95% 以上。

## 8. 风险控制

| 风险 | 处理方式 |
| --- | --- |
| 规则范围膨胀 | 只落地 RSA、TLS、弱算法 P0 规则，其余进入 P1 文档化 |
| PDF 解析不稳定 | 先支持原生 PDF 和简单图片抽取，复杂扫描件留到 S2 |
| 文档和代码不一致 | 每个接口 PR 必须同时更新 stage-interface 文档 |
| 团队并行冲突 | 小 PR，按模块评审，禁止跨模块大改 |

## 9. 退出标准

S1 完成后，应留下一个可继续开发的工程基座，而不是一次性 demo。退出时必须具备：
- 可运行测试集。
- 明确的 S2 输入字段要求。
- 首版标准条款、检查项与结果映射边界。
- 最小 parser/model/rules/web 链路。
- 已知风险和下阶段依赖清单。
