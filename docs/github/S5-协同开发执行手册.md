# S5 协同开发执行手册（2026-07-11 至 2026-07-31）

## 1. 阶段定位

S5 是发布收敛和结题交付阶段。目标不是继续堆功能，而是冻结版本、修阻断缺陷、补齐文档、完成演示和答辩材料。

S5 的成功标准是“可运行、可复现、可讲清、可归档”。

## 2. 入口条件

- S4 主流程已完成彩排。
- 发布候选风险清单已列出。
- 至少一份示例报告可完整跑通。
- 测试、部署、用户手册责任人明确。

## 3. 范围边界

必须完成：
- 发布候选分支。
- 最终回归测试。
- 用户手册。
- 部署说明。
- 演示脚本和答辩材料。
- 示例报告和导出样例。
- 版本归档和最终验收纪要。

明确不做：
- 不新增高风险功能。
- 不重构核心链路。
- 不调整已冻结接口。
- 不引入新外部依赖，除非解决阻断问题。

## 4. 发布策略

分支策略：
- 从稳定集成分支创建 `release/v1.0-final`。
- 只允许阻断缺陷、文档、演示脚本、发布包相关变更进入 release。
- 非阻断优化记录到后续版本，不进入 S5 发布范围。

版本策略：
- 07-20：文档初版齐套。
- 07-25：发布候选冻结。
- 07-28：最终演示彩排。
- 07-31：发布 `v1.0-final` 并归档。

## 6. 发布检查清单

发布前必须确认：
- `pytest -q` 通过。
- 端到端演示脚本通过。
- 示例报告可跑通。
- 导出文件可打开，字段和页面一致。
- 部署说明可按步骤复现。
- 用户手册覆盖上传、查看、筛选、证据、导出。
- 答辩材料能解释系统边界和后续演进。

## 7. 质量门禁

S5 合并门禁：
- release 分支只允许小 PR。
- 每个 PR 必须说明是否影响演示路径。
- 发布相关变更至少 2 人评审。
- 阻断缺陷必须当天闭环。

M5 验收必须满足：
- 核心流程可演示且可复现。
- 结题文档齐套并版本一致。
- 发布回归通过率不低于 95%。
- release 分支无阻断缺陷。
- `v1.0-final` 标签已发布。

## 8. 风险控制

| 风险 | 处理方式 |
| --- | --- |
| 临近交付范围膨胀 | 只允许进入发布风险清单，不进入 release |
| 文档和代码版本不一致 | 每日核对提交号、版本号和演示脚本 |
| 发布候选缺陷未收敛 | 阻断缺陷 P0 日清，非阻断缺陷延期 |
| 演示环境不可复现 | 保留固定样本、固定命令、固定导出结果 |

## 9. 退出标准

S5 完成后必须归档：
- 发布标签和提交号。
- 可运行源码和依赖说明。
- 示例报告和导出结果。
- 测试报告。
- 用户手册和部署说明。
- 答辩材料。
- 最终验收纪要。
