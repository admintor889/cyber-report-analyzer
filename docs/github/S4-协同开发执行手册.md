# S4 协同开发执行手册（2026-06-21 至 2026-07-10）

## 1. 阶段定位

S4 是 Web 集成和规则维护阶段。目标是把 S3 的分析结果接入轻量 Web 流程，让用户能完成上传、查看、筛选、证据查看、规则维护和导出，并保证页面与导出的条款级结果一致。

S4 的成功标准是“主流程可演示、规则可维护、导出和页面一致”。

## 2. 入口条件

- S3 主链路结果结构已冻结。
- `RuleResult`、`ReviewItem`、`EvidenceTrace` 可稳定生成。
- 至少有一组可演示样本。
- 规则和模型回归集可运行。
- Web 规则维护接口和数据模型已确定。

## 3. 范围边界

必须完成：
- 轻量上传和任务查询流程。
- 结果列表、筛选、详情查看。
- 证据查看。
- 规则维护界面和立即生效流程。
- JSON/CSV 或自定义格式导出。
- 误判样本收集和修正规则。
- 主流程端到端回归。
- PDF 和 Word 上传在页面上的展示口径一致，不能因为输入格式不同而分裂成两套页面逻辑。

明确不做：
- 不做复杂权限角色。
- 不做企业级多租户。
- 不做大屏展示。
- 不做高并发部署。

## 4. 演示主流程

S4 演示路径必须固定：
1. 上传或选择样本报告。
2. 查看任务状态。
3. 查看 summary。
4. 按 `FAIL/REVIEW` 筛选问题项。
5. 打开某条结果的证据页码、片段或截图。
6. 在 Web 中调整一条规则并保存。
7. 刷新结果，确认规则维护立即生效。
8. 导出结果。
9. 展示导出内容和页面内容一致。

Word 样本演示要求：
- 至少保留一份 Word 样本作为回归或答辩备用样例。
- Word 文档的页面结果、证据与导出结果必须和 PDF 样本遵守同一字段语义。

- 06-30：S4 端到端回归基线建立。
- 07-03：上传、查询、证据查看可串联。

- 07-08：规则维护、导出和误判修正完成。
- 07-10：M4 验收。

## 7. 质量门禁

每个 S4 PR 必须满足：
- 交互或展示变更必须附截图、录屏或接口响应样例。
- 缺陷修复必须包含复现步骤和回归结果。
- 导出字段必须和页面展示字段一致。
- Web 返回结构不能破坏 S3 核心字段。
- 页面上的 `FAIL` 只表示条款不符合，不表示整份报告失败。
- 规则维护保存后必须有版本号或生效时间记录。

M4 验收必须满足：
- 上传到导出主流程可演示。
- 关键问题项可查看证据来源。
- 规则维护立即生效闭环可复现。
- S4 回归通过率不低于 92%。
- CI 持续绿灯。

## 8. 风险控制

| 风险 | 处理方式 |
| --- | --- |
| 联调周期过长 | 先做固定演示路径，再补非核心功能 |
| 页面字段和后端不一致 | 建立字段对齐表，每天同步 |
| 导出与页面不一致 | 导出直接复用页面 view model |
| 规则维护结果不生效 | 规则保存后立即做回读校验 |
| 缺陷反复出现 | 高频缺陷升级为 P0，必须附回归用例 |

## 9. 退出标准

S4 完成后必须交给 S5：
- 可运行演示路径。
- 可导出的示例结果。
- 缺陷闭环清单。
- 部署和联调说明。
- 发布候选风险清单。
- 规则维护操作记录和最终冻结版本号。
