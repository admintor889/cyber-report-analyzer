# S3 协同开发执行手册（2026-05-21 至 2026-06-20）

## 1. 阶段定位

S3 是规则融合和条款级判定阶段。目标是把 S2 输出的结构化字段接入主规则库，完成两份清单的统一建模、条款级 `PASS/FAIL/REVIEW` 判定、复核解释和证据追溯，形成可解释、可回放的分析结果。

S3 的重点不是界面，而是规则可信、复核可控、证据链不断、规则维护有版本。

## 2. 入口条件

- S2 已输出稳定的 `StructuredField[]`。
- 字段映射表和输入类型定义已冻结。
- 两份主规则来源已整理为可合并的源数据。
- 至少 5 份样本可以从 PDF/Word 处理到结构化字段。
- parser/OCR 失败样本已有记录。

## 3. 范围边界

必须完成：
- 主规则库数据结构和版本化定义。
- 两份清单合并后的规则集和三态判定。
- 待复核项分类。
- 模型批量复核和解释输出。
- 证据追溯结构。
- 简要分析报告和 findings 聚合。
- 规则维护的后端契约与持久化。
- Word 输入的条款命中与 PDF 输入保持同一套规则判定口径。

明确不做：
- 不做复杂权限。
- 不做完整前端交互。
- 不做模型训练。
- 不做全量企业规则库。

## 4. 核心判定原则

- `PASS`：字段存在、格式可信、满足规则阈值。
- `FAIL`：字段存在、格式可信、明确违反规则。
- `REVIEW`：字段缺失、低置信度、语义模糊、规则冲突或命中弱算法关键词。
- 任一 `FAIL` 只表示对应标准条款不符合，不表示整份报告失败。
- `summary` 只做聚合统计，不做全局裁决。

规则优先级：
- P0：必须执行，影响 M3 验收。
- P1：优先实现，未完成必须说明原因。
- P2：可文档化，不阻塞 M3。

模型职责：
- 归一化表达。
- 解释模糊项。
- 给出复核建议。
- 不直接覆盖规则结论。

## 6. 联调主线

S3 主链路：
1. 读取 S2 的结构化字段，字段来源可以是 PDF、Word 或 OCR。
2. 加载由两份清单合并后的主规则库。
3. rules_engine 输出 `RuleResult[]`。
4. classify_review_items 筛出待复核项。
5. model_review 批量输出复核解释。
6. evidence 绑定页码、片段、截图。
7. reporting 输出 summary、findings 和 highlights。
8. web 回填最终结果和规则维护日志。

联调检查点：
- 05-26：规则优先级和冲突策略冻结。
- 06-05：规则引擎和模型复核分别可用。
- 06-10：证据追溯结构可用。
- 06-15：提取、判定、复核、证据链路预验收。
- 06-20：M3 验收。

## 7. 质量门禁

每个 S3 PR 必须满足：
- 规则变更必须有命中和未命中测试。
- REVIEW 触发策略必须有样例。
- 模型复核结果不能覆盖规则 verdict。
- 证据结构必须保留 `field/value/page/snippet/source_type`。
- 对外结果字段变更必须同步接口文档。
- 条款编号、检查项名称与主规则来源必须同步维护。

M3 验收必须满足：
- 提取、判定、复核主链路跑通。
- 三态结果稳定。
- 待复核项可追踪。
- 结果可关联页码、片段、截图来源。
- 主规则库版本和来源映射可回放。
- S3 回归通过率不低于 90%。

## 8. 风险控制

| 风险 | 处理方式 |
| --- | --- |
| 规则与模型结论冲突 | 规则 verdict 不被模型覆盖，模型只写 explanation 和 suggestion |
| 待复核数量过多 | 建立 REVIEW 原因分类，优先修复字段缺失和低置信度 |
| 证据链断裂 | 没有证据的 FAIL 必须降级为 REVIEW |
| 规则误判 | 所有误判进入样本集，并在 S4 做命中率优化 |

## 9. 退出标准

S3 完成后必须交给 S4：
- 稳定的结果结构。
- 可展示的证据结构。
- 待复核项列表。
- 简要分析报告数据。
- 规则和模型回归集。
- 主规则库版本说明和维护记录。
- Word 样本与 PDF 样本在结果结构上没有分叉，页面、导出和证据追溯口径一致。
