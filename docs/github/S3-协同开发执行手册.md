# S3 协同开发执行手册（条款级 finding 版）

## 1. 阶段定位

S3 是规则融合和条款级判定阶段。目标是把 S2 输出的结构化字段接入主规则库，完成两份清单的统一建模、条款级判定、复核解释和证据追溯。

## 2. 入口条件

- S2 已输出稳定的 `StructuredField[]`。
- 字段映射表和输入类型定义已冻结。
- 两份主规则来源已整理为可合并的源数据。
- 至少 5 份样本可以从 PDF/Word 处理到结构化字段。

## 3. 范围边界

必须完成：
- 主规则库数据结构和版本化定义。
- 两份清单合并后的规则集和三态判定。
- 待复核项分类。
- 模型批量复核和解释输出。
- 证据追溯结构。
- 简要分析报告和 findings 聚合。
- 规则维护的后端契约与持久化。

明确不做：
- 不做复杂权限。
- 不做完整前端交互。
- 不做模型训练。
- 不做全量企业规则库。

## 4. 核心判定原则

- `PASS`：字段存在、格式可信、满足规则阈值。
- `FAIL`：字段存在、格式可信、明确违反规则。
- `REVIEW`：字段缺失、低置信度、语义模糊、规则冲突或命中弱算法关键词。
- 任一 `FAIL` 只表示对应标准条款不符合，不表示整份报告失败。
- `summary` 只做聚合统计，不做全局裁决。

## 5. 联调主线

1. 读取 S2 的结构化字段。
2. 加载主规则库。
3. rules_engine 输出 `RuleResult[]`。
4. classify_review_items 筛出待复核项。
5. model_review 批量输出复核解释。
6. evidence 绑定页码、段落编号、片段和截图来源。
7. reporting 输出 summary、findings 和 highlights。
8. web 回填最终结果和规则维护日志。

## 6. 质量门禁

- 规则变更必须有命中和未命中测试。
- REVIEW 触发策略必须有样例。
- 模型复核结果不能覆盖规则 verdict。
- 证据结构必须保留 `field/value/page/section_id/paragraph_id/snippet/source_type`。
- 对外结果字段变更必须同步接口文档。

## 7. 退出标准

S3 完成后必须交给 S4 稳定的结果结构、可展示的证据结构、待复核项列表、简要分析报告数据和规则回归集。
