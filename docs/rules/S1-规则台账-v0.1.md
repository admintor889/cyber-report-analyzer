# S1 规则台账 v0.1

## 1. 说明

本台账用于第一阶段规则梳理，目标是先确定高频可判定条款，并把条款输出到“检查项”级别。

本台账不定义整份报告的一票否决规则。S1 的输出单位始终是标准条款或检查项，而不是整份报告。

## 2. P0 规则（S1 必须落地）

### 2.1 RSA 密钥长度识别

- 识别目标：`RSA-2048`、`RSA-3072` 等
- 对应标准口径：`标准1.1-密钥长度`
- 初步判定：
  - `>=3072` 倾向 `PASS`
  - `2048` 倾向 `REVIEW`
  - `<2048` 倾向 `FAIL`
- 输出示例：
  - `标准1.1-密钥长度-RSA2048`
  - `标准1.1-密钥长度-需人工检查`

### 2.2 TLS 协议版本识别

- 识别目标：`TLS1.0`、`TLS1.1`、`TLS1.2`、`TLS1.3`
- 对应标准口径：`标准1.2-TLS协议版本`
- 初步判定：
  - `1.2/1.3` 倾向 `PASS`
  - `1.0/1.1` 倾向 `REVIEW` 或 `FAIL`
- 输出示例：
  - `标准1.2-TLS协议版本-TLS1.2`
  - `标准1.2-TLS协议版本-内容和截图冲突`

### 2.3 弱算法关键词识别

- 识别目标：`MD5`、`SHA-1`、`DES`、`3DES`、`RC4`、`ECB`
- 对应标准口径：`标准1.2-不合适的加密算法`
- 初步判定：
  - 命中即进入 `FAIL` 或 `REVIEW`
  - 若文本不足以完全确认，输出 `需人工检查`
- 输出示例：
  - `标准1.2-不合适的加密算法-具体使用的加密算法`
  - `标准1.2-不合适的加密算法-内容和截图冲突`

## 3. P1 规则（S1 文档化，后续版本扩展）

1. 端口服务与不安全配置关键词
2. 更新机制与签名机制检查项
3. 认证方式与默认口令相关条目

## 4. 判定口径

- `PASS`：检查项满足对应标准条款。
- `FAIL`：检查项不满足对应标准条款，输出不符合项。
- `REVIEW`：缺少足够证据、内容与截图冲突、或措辞不足以判断。
- `deviation`：S1 可忽略，不作为实现重点。

## 5. 字段标准化建议

- `field`: 规则字段名，如 `crypto.rsa.key_length`
- `value`: 提取值，如 `2048`
- `source_type`: `text` / `image_ocr`
- `page`: 来源页码
- `snippet`: 证据片段

## 6. S1 输出要求

- 规则台账必须可追踪到标准表和检查项来源。
- 每条规则必须定义：识别模式、判定倾向、复核策略。
- 每条规则必须标注优先级（P0/P1）。
- 不得将单条 `FAIL` 解释为整份报告 `FAIL`。
