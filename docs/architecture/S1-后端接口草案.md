# S1 后端接口草案

## 1. 目标

定义第一阶段最小可运行后端接口，支持任务提交、任务查询、结果查询、待复核查询和结果回填。

S1 的后端结果单位是标准条款 / 检查项级别，不提供整份报告的一票否决结果。

## 2. 接口列表（函数级草案）

1. `health()`
- 说明：健康检查
- 返回：`{"status": "ok"}`

2. `submit_report(file_name: str)`
- 说明：提交报告任务
- 输入：文件名字符串
- 返回：任务对象

3. `get_task(task_id: str)`
- 说明：查询任务
- 输入：任务 ID
- 返回：任务对象（不存在抛出 `KeyError`）

4. `set_task_result(task_id: str, result: dict)`
- 说明：回填分析结果并更新任务状态
- 输入：任务 ID、结果对象
- 返回：更新后的任务对象

5. `get_task_result(task_id: str)`
- 说明：查询任务结果对象
- 输入：任务 ID
- 返回：结果对象或 `null`

6. `get_pending_reviews(task_id: str)`
- 说明：查询待复核条目列表
- 输入：任务 ID
- 返回：待复核列表

## 3. 任务对象结构

```json
{
  "task_id": "task-0001",
  "file_name": "sample-report.pdf",
  "status": "queued | completed",
  "result": null
}
```

## 4. 结果对象结构

```json
{
  "schema_version": "s1.v1",
  "task_id": "task-0001",
  "fields": {
    "crypto.rsa.key_length": "2048"
  },
  "rule_results": [
    {
      "rule_id": "S1-RSA-001",
      "standard_id": "S1-1.2",
      "check_item": "不合适的加密算法",
      "field": "crypto.rsa.key_length",
      "value": "2048",
      "verdict": "REVIEW",
      "reason": "RSA 2048 requires policy review in S1 baseline."
    }
  ],
  "pending_reviews": [],
  "review_details": [],
  "summary": {
    "PASS": 1,
    "FAIL": 0,
    "REVIEW": 2
  }
}
```

## 5. 标准与检查项数据约束

- 标准表由 UI 后台录入，不直接硬编码在接口层。
- 检查项可以关联一条或多条标准。
- 接口返回的 findings 必须能追溯到标准编号、检查项名称和证据片段。
- `FAIL` 只表示对应检查项不符合，不表示整份报告失败。

## 6. S1 约束

- 先采用内存存储，不引入数据库依赖。
- 不做鉴权和权限控制。
- 不做并发和超时控制。
- 待复核接口仅返回内存中的 `REVIEW` 项，不引入独立工作流。

## 7. S2 演进方向

- 接入 SQLite 存储任务与结果。
- 增加任务状态机（queued/running/completed/failed）。
- 增加标准表、检查项和映射关系的持久化接口。
- 增加分页查询和筛选接口。
