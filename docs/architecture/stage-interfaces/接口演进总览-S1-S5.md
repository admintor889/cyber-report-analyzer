## 2. 核心接口演进表

### 2.1 web

- S2: 任务创建、状态查询、结构化字段查询和输入类型记录，支撑 PDF/Word 解析与 OCR 回填
- S3: 接入规则、复核、证据结果回填，形成条款级判定主链路，并支持主规则来源的版本化读取
- S4: 增加 list_results/get_evidence/export_result 和规则维护相关接口，支撑页面、导出和 Web 即时生效的一致性
- S5: 统一 Envelope 响应、版本化查询、发布导出和归档信息

### 2.2 parser + ocr

- S2: 输出 ParseResultV2、OCRResultV2、StructuredField，并保留 PDF 与 Word 的来源类型
- S3: StructuredField 适配 rules 输入格式，保证页码、片段、置信度和来源引用可回溯
- S4: 保持字段语义稳定并优化联调一致性
- S5: 冻结为发布版稳定输入源

### 2.3 rules + model + evidence

- S2: 完成字段规范化、来源保存和低置信度标记，为主规则库输入做准备
- S3: 两份清单合并后的主规则集、冲突处理、条款级判定、review 复核和 evidence 追溯主链路打通
- S4: 规则维护结果、追溯结果可视化、导出一致性
- S5: 规则与证据接口冻结用于发布

### 2.4 reporting + storage

- S2: 增加 parse/ocr/field 保存和回放语义
- S3: 增加 findings、summary 和分析报告结构
- S4: 增加详情导出结构和页面 view model
- S5: 增加 release 报告与 artifact 信息

## 3. 字段稳定性分级

- A级（不得破坏）
  - task_id
  - verdict
  - summary.PASS/FAIL/REVIEW
  - evidence.page/snippet/image_ref

- B级（可扩展不可删除）
  - confidence
  - reason
  - meta
  - source_type
  - rule_version

- C级（阶段内可调整）
  - 中间调试字段
  - 临时统计字段

## 4. 团队执行建议

- 每阶段开始前先冻结接口文档中的 A 级字段。
- 每次接口变更必须更新对应阶段文档。
- PR 描述中必须注明“接口是否破坏兼容”。
- S2 结束前必须冻结 StructuredField，否则 S3 规则引擎不能稳定开发。
- S3 结束前必须冻结 RuleResult、EvidenceTrace 和主规则来源映射，否则 S4 页面和导出会反复返工。
- S4 结束前必须冻结导出格式和规则维护接口，否则 S5 发布文档无法收敛。

## 5. 阶段统一口径

- 任一单条 `FAIL` 只表示对应标准条款或检查项不符合，不表示整份报告失败。
- `summary` 只是条款级结果的聚合统计，不是全局裁决。
- `REVIEW` 用于低置信度、语义歧义、证据不足或需要人工确认的项。

## 6. 阶段接口冻结点

| 阶段 | 必须冻结的接口或字段 | 影响下游 |
| --- | --- | --- |
| S2 | ParseResultV2、OCRResultV2、StructuredField、input_type | S3 规则输入和证据追溯 |
| S3 | RuleDefinition、RuleResult、ReviewItem、EvidenceTraceV3、findings view model | S4 页面展示、规则维护和导出 |
| S4 | ResultViewItem、ExportPayload、rule admin payload、导出详情结构 | S5 发布包和用户手册 |
| S5 | Envelope、ErrorObject、发布版导出 artifact | 结题归档和后续版本维护 |
