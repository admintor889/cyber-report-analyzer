## 2. 核心接口演进表

### 2.1 web

- S1: health/submit_report/get_task/set_task_result（最小闭环）
- S2: 增强任务状态和任务元信息
- S3: 接入规则与复核结果回填
- S4: 增加 list_results/get_evidence/export_result
- S5: 统一 Envelope 响应与版本化查询导出

### 2.2 parser + ocr

- S1: 占位接口
- S2: 输出 ParseResultV2/OCRResultV2，支持结构化字段
- S3: 适配 rules 输入格式
- S4: 保持稳定并优化联调一致性
- S5: 作为发布版稳定输入源

### 2.3 rules + model + evidence

- S1: model_review 可用，rules/evidence 占位
- S2: 先完成字段规范化与保存
- S3: rules 判定 + review 复核 + evidence 追溯主链路打通
- S4: 追溯结果可视化、导出一致性
- S5: 规则与证据接口冻结用于发布

### 2.4 reporting + storage

- S1: summary 统计 + 内存存储
- S2: 增加 parse/ocr/field 持久化语义
- S3: 增加分析报告结构
- S4: 增加详情导出结构
- S5: 增加 release 报告与 artifact 信息

## 3. 字段稳定性分级

- A级（不得破坏）
  - task_id
  - verdict
  - summary.PASS/FAIL/REVIEW
  - evidence.page/snippet/image_ref

- B级（可扩展不可删除）
  - confidence
  - reason
  - meta

- C级（阶段内可调整）
  - 中间调试字段
  - 临时统计字段

## 4. 团队执行建议

- 每阶段开始前先冻结接口文档中的 A 级字段。
- 每次接口变更必须更新对应阶段文档。
- PR 描述中必须注明“接口是否破坏兼容”。
