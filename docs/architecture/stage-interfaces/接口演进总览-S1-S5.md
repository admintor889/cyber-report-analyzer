## 2. 核心接口演进表

### 2.1 web

- S1: health/submit_report/get_task/set_task_result/get_task_result/get_pending_reviews（最小任务闭环，输出条款级 findings）
- S2: 增强任务状态、任务元信息和结构化字段查询
- S3: 接入规则、复核、证据结果回填，形成条款级判定主链路
- S4: 增加 list_results/get_evidence/export_result，支撑页面和导出一致性
- S5: 统一 Envelope 响应、版本化查询和发布导出

### 2.2 parser + ocr

- S1: parser MVP，OCR 占位和后处理策略
- S2: 输出 ParseResultV2/OCRResultV2/StructuredField
- S3: StructuredField 适配 rules 输入格式
- S4: 保持字段语义稳定并优化联调一致性
- S5: 冻结为发布版稳定输入源

### 2.3 rules + model + evidence

- S1: 标准表、检查项、条款级结果映射；model_review 最小归一化；evidence 数据结构草案
- S2: 完成字段规范化、来源保存和低置信度标记
- S3: rules 判定 + review 复核 + evidence 追溯主链路打通
- S4: 追溯结果可视化、导出一致性
- S5: 规则与证据接口冻结用于发布

### 2.4 reporting + storage

- S1: summary 统计 + 内存存储
- S2: 增加 parse/ocr/field 保存和回放语义
- S3: 增加分析报告结构
- S4: 增加详情导出结构
- S5: 增加 release 报告与 artifact 信息

## 3. 字段稳定性分级

- A级（不得破坏）
  - task_id
  - verdict
  - summary.PASS/FAIL/REVIEW
  - evidence.page/snippet/image_ref

- B级（可扩展不可删除）
  - confidence
  - reason
  - meta

- C级（阶段内可调整）
  - 中间调试字段
  - 临时统计字段

## 4. 团队执行建议

- 每阶段开始前先冻结接口文档中的 A 级字段。
- 每次接口变更必须更新对应阶段文档。
- PR 描述中必须注明“接口是否破坏兼容”。
- S2 结束前必须冻结 StructuredField，否则 S3 规则引擎不能稳定开发。
- S3 结束前必须冻结 RuleResult 和 EvidenceTrace，否则 S4 页面和导出会反复返工。
- S4 结束前必须冻结导出格式，否则 S5 发布文档无法收敛。

## 5. 阶段统一口径

- 任一单条 `FAIL` 只表示对应标准条款或检查项不符合，不表示整份报告失败。
- `summary` 只是条款级结果的聚合统计，不是全局裁决。
- `REVIEW` 用于低置信度、语义歧义、证据不足或需要人工确认的项。

## 6. 阶段接口冻结点

| 阶段 | 必须冻结的接口或字段 | 影响下游 |
| --- | --- | --- |
| S1 | Task、SemanticReviewResult、基础 RuleResult、summary、findings | S2 联调和最小后端任务流 |
| S2 | ParseResultV2、OCRResultV2、StructuredField | S3 规则输入和证据追溯 |
| S3 | RuleDefinition、RuleResult、ReviewItem、EvidenceTraceV3、finding view model | S4 页面展示和导出 |
| S4 | ResultViewItem、ExportPayload、导出详情结构 | S5 发布包和用户手册 |
| S5 | Envelope、ErrorObject、发布版导出 artifact | 结题归档和后续版本维护 |
