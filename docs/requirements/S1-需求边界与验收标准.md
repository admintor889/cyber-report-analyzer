# S1 需求边界与验收标准

## 1. 文档目的

定义第一阶段（S1）的开发边界、判定口径和验收标准，避免把单条检查结果误当成整份报告结论。

S1 的设计原则是：按标准条款和检查项输出不符合项，单条 Fail 只影响对应条款，不直接导致整份报告 Fail。

## 2. S1 范围内必须完成

- 完成标准表与检查项的 UI 后台录入能力。
- 完成检查项与标准条款的多对多映射。
- 完成 PDF 检查的条款级输出，不使用报告级一票否决。
- 完成解析、抽取、归一化、结果回填的最小闭环。
- 完成基础权限管理，保证管理员可维护标准和检查项。
- 完成模型归一化协议与后端接口草案。
- 完成字段映射、样本设计和回归模板文档。

## 3. S1 范围外内容

- 不做报告级单点一票否决机制。
- 不做 deviation / Not Applicable 的完整自动化处理。
- 不做高并发性能优化与集群部署。
- 不做模型训练与复杂效果调优。
- 不做超出原型范围的企业级多租户权限体系。

## 4. 输入与输出定义

输入：

- 样本 PDF 报告（原生 PDF 优先）
- 标准表数据
- 检查项数据
- 检查项与标准条款映射关系
- 截图样本与证据片段

输出：

- 协同文档（执行手册、Issue 清单、验收清单）
- 标准表与检查项映射规范
- 模型协议与后端接口草案
- 条款级不符合项列表
- 可运行源码骨架与基础测试

## 5. S1 验收标准（DoD）

- 标准表、检查项和映射关系可通过 UI 录入和维护。
- UI / UE 按原型设计执行。
- 基础权限管理可用，管理员和普通使用者访问边界清晰。
- PDF 检查输出为条款级不符合项，不存在单条规则导致整份报告 Fail 的逻辑。
- 检查正确率达到 95% 以上。
- `develop` 或当前集成分支可通过 CI。
- 文档、代码、测试三者版本一致。
- 里程碑标签 `v0.1-m1` 可按计划发布。

## 6. 结果口径

- `PASS`：符合标准条款。
- `FAIL`：不符合标准条款，形成不符合项。
- `REVIEW`：缺少足够证据、正文和截图冲突、或需人工检查。
- `deviation`：S1 可忽略，后续版本再考虑映射为 Not Applicable。

## 7. 风险与控制

- 风险：检查项和标准条款映射不一致。
- 控制：每次录入标准或检查项时同步更新映射文档和样本用例。
- 风险：正文与截图证据冲突。
- 控制：统一输出 REVIEW 型结果，并保留证据片段。
- 风险：条款级输出和报告级结论混淆。
- 控制：验收时只看条款级不符合项，不看全局 Fail 标签。
