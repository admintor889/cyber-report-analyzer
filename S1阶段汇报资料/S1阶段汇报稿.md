本阶段完成的主要功能包括：

## 二、模块设计与职责划分

### 1. parser 模块

parser 负责从 PDF 中提取基础内容，是整条链路的起点。它的职责不是做复杂版面理解，而是先把报告拆成可供下游使用的两类信息：正文文本块和图片文件。

在当前实现中，parser 主要承担以下任务：

- 识别 PDF 流对象并提取文本片段。
- 导出嵌入图片，供 OCR 或证据回看使用。
- 输出统一结构，方便后续模块直接消费。

它解决的是“原始 PDF 如何变成可处理数据”的问题。

关键代码节选：

```python
def extract_text_and_images(pdf_path: Path) -> Dict[str, List[str]]:
	# 先做输入校验，避免把非法路径带入后续解析。
	if not isinstance(pdf_path, Path):
		raise TypeError("pdf_path must be a pathlib.Path")
	if not pdf_path.exists():
		raise FileNotFoundError(f"pdf not found: {pdf_path}")

	# parser 只负责拆解，不做规则判定。
	pdf_bytes = pdf_path.read_bytes()
	text_blocks = _extract_pdf_text_blocks(pdf_bytes)
	image_paths = _extract_pdf_images(pdf_bytes, pdf_path)
	return {"text_blocks": text_blocks, "image_paths": image_paths}
```

### 2. rules_engine 模块

rules_engine 负责把抽取到的字段映射到 S1 规则口径，并输出条款级结果。它是整个项目里最核心的判定层。

当前模块承担的职责包括：

- 对字段和规则进行基础匹配。
- 支持数值比较、字符串比较和关键词命中。
- 输出条款级结果，统一为 `PASS / FAIL / REVIEW`。
- 当字段缺失、规则不完整或证据不足时，自动进入 `REVIEW`。

这个模块强调的是“检查项是否成立”，而不是“整份报告是否一票否决”。

关键代码节选：

```python
def evaluate_s1_baseline(fields: Dict[str, str]) -> List[Dict[str, str]]:
	# S1 只冻结 P0 规则，结果始终是条款级 verdict。
	rsa_raw = str(fields.get("crypto.rsa.key_length", "")).strip()
	if not rsa_raw:
		results.append(
			_build_rule_result(
				rsa_rule,
				field="crypto.rsa.key_length",
				value="",
				verdict="REVIEW",
				reason="RSA key length is missing.",
			)
		)
	else:
		rsa_bits = int(float(rsa_raw))
		if rsa_bits >= 3072:
			verdict = "PASS"
		elif rsa_bits == 2048:
			verdict = "REVIEW"
		else:
			verdict = "FAIL"
```

### 3. model_review 模块

model_review 负责语义归一化和人工复核前的预处理。它不是模型训练模块，而是一个轻量的语义标准化层。

当前模块的主要作用是：

- 把 RSA2048、2048-bit RSA、TLS1.2 这类表达统一成标准化写法。
- 对弱算法关键词做归一化标记。
- 给出解释信息、置信度和策略提示，便于后续复核或分流。

它解决的是“同一个安全概念在报告里有很多写法，如何统一表达”的问题。

关键代码节选：

```python
def semantic_review(item: Dict[str, str]) -> Dict[str, str]:
	# 先拼接字段、规则和原文上下文，提升数值字段的识别准确率。
	context = " ".join(
		[
			field,
			str(item.get("rule_id") or ""),
			str(item.get("reason") or ""),
			lower,
		]
	).lower()

	# 数值本身不够时，结合上下文补全 RSA / TLS 语义。
	if ("rsa" in context or "key_length" in context) and re.fullmatch(r"\d{3,4}", text):
		return semantic_review({"value": f"RSA{text}"})
```

### 5. reporting 模块

reporting 负责把分析结果汇总成便于展示和汇报的输出，包括 PASS / FAIL / REVIEW 统计、告警信息和最终 JSON 结果。

它的作用是把前面几个模块的输出收拢成一份可以直接交付或展示的结果。

关键代码节选：

```python
def export_summary(results: List[Dict[str, str]]) -> Dict[str, int]:
	# 只统计三种条款级结果，未知 verdict 统一记入 REVIEW。
	summary = {"PASS": 0, "FAIL": 0, "REVIEW": 0}
	for item in results:
		verdict = item.get("verdict", "REVIEW")
		if verdict not in summary:
			verdict = "REVIEW"
		summary[verdict] += 1
	return summary
```

### 6. web 模块

web 模块当前承担的是任务流转和结果回填功能，属于后端任务骨架。它用内存任务表模拟了提交、查询、完成和待复核读取等行为。

这部分的意义在于：即使还没有完整数据库和服务框架，整个流程也能先跑出一个任务闭环，方便后续扩展成真正的后台服务。

关键代码节选：

```python
def analyze_task(task_id: str, fields: Dict[str, str], rules: List[Dict[str, str]]) -> Dict[str, Any]:
	# 先做规则判定，再生成待复核清单和汇总信息。
	if rules:
		rule_results = evaluate_rules(fields, rules)
	else:
		rule_results = evaluate_s1_baseline(fields)

	pending_reviews = classify_review_items(rule_results)
	normalized_reviews = batch_semantic_review(pending_reviews)
	review_details: List[Dict[str, str]] = []
	for pending, normalized in zip(pending_reviews, normalized_reviews):
		review_details.append({**pending, **normalized})

	payload: Dict[str, Any] = {
		"schema_version": "s1.v1",
		"task_id": task_id,
		"fields": fields,
		"rule_results": rule_results,
		"pending_reviews": pending_reviews,
		"review_details": review_details,
		"summary": export_summary(rule_results),
	}
```

## 三、关键口径补充

这部分是本阶段最容易混淆、但已经明确统一的边界，汇报时建议直接说清楚。

- 单条规则 `FAIL` 不会导致整份报告 `FAIL`，只影响对应的标准条款或检查项。
- 实际输出单位是“标准条款 + 检查项”级别，不是整份报告级别。
- 检查项由管理员在 UI 后台录入，并且录入时必须关联一个或多个标准条款。
- 最终不符合项要按标准条款和检查项名称输出，例如：`标准1.2-不合适的加密算法-具体使用的加密算法`。
- 正文、截图、OCR 三者如果出现不一致，直接按严重冲突处理，结果判定为 `FAIL`，不进入正常通过链路。
- 结果展示时要同时保留原始命中的文本片段和归一化后的值，便于人工复核和溯源。
- `deviation` 在 S1 阶段可先忽略，实际开发中按 `PASS / FAIL / REVIEW` 三态处理即可。
- 缺少足够话术但还能通过检查项判断时，输出 `需人工检查`。
- 正文和截图冲突时，输出 `内容和截图冲突`。

如果后续能拿到标准表和 checklist 的原始 xlsx / docx，还可以把标准编号、检查项名称和映射关系进一步补成后台录入示例。

## 三、实机演示流程

本次演示的目标是展示一个 PDF 被完整解析、抽取、判定并汇总的全过程。实际演示材料已经按步骤截图，适合在汇报时按“输入 -> 抽取 -> 判定 -> 复核 -> 汇总”的顺序讲解。

### 步骤 1：PDF 解析

先运行 parser，对原始 PDF 做基础拆解，输出文本块和图片信息。

```bash
python src\parser\pdf_parser.py "tests\网络空间安全基地+南昌大学学工一体化平台垂直越权+攻击报告.pdf" --output outputs\s1_demo\01_parse.json
```

这一阶段可以看到，系统成功把报告拆成了大量文本块，并识别出 3 张图片，为后续字段抽取和证据追溯打下基础。

### 步骤 2：字段抽取

随后运行字段抽取器，把 parser 输出中的内容进一步整理成结构化字段。

示例命令：

```bash
python src\rules_engine\s1_field_extractor.py --parse-file outputs\s1_demo\01_parse.json --output outputs\s1_demo\02_fields.json
```

![](S1%E9%98%B6%E6%AE%B5%E6%B1%87%E6%8A%A5%E7%A8%BF_images/image-20260420193953210.png)

这一阶段会生成 `fields`，并保留 `raw_text` 作为下游规则和复核的输入。当前样例里能够抽出弱算法相关字段，但 RSA 和 TLS 字段并不总是稳定命中，因此后续结果会保留 REVIEW 口径。

### 步骤 3：模型复核与归一化

接着对抽取到的字段做语义归一化，统一表达形式，便于人工检查和规则判定。

示例命令：

```bash
python src\model_review\reviewer.py --fields-file outputs\s1_demo\02_fields.json --output outputs\s1_demo\03_review.json
```

![](S1%E9%98%B6%E6%AE%B5%E6%B1%87%E6%8A%A5%E7%A8%BF_images/image-20260420193941582.png)

这一阶段会把弱算法关键词转换成标准化表达，例如将 `DES,RC4` 归一成 `WEAK-ALGO-DES` 这类结果，同时给出解释、置信度和策略提示。

### 步骤 4：规则判定

在字段和复核结果准备好之后，运行规则引擎做条款级判定。

```bash
python src\rules_engine\rule_engine.py --fields-file outputs\s1_demo\02_fields.json --output outputs\s1_demo\04_rules.json
```

![](S1%E9%98%B6%E6%AE%B5%E6%B1%87%E6%8A%A5%E7%A8%BF_images/image-20260420193932596.png)

这一步输出的是条款级规则结果，而不是整份报告的总失败结论。演示样例最终显示为 `PASS 0 / FAIL 0 / REVIEW 3`，说明流程跑通了，但样例报告里关键字段证据仍需要人工确认。

### 步骤 5：任务回填

规则结果生成后，再把字段文件送入 web 任务层，模拟提交任务和结果回填。

```bash
python src\web\app.py --fields-file outputs\s1_demo\02_fields.json --output outputs\s1_demo\05_task.json
```

![](S1%E9%98%B6%E6%AE%B5%E6%B1%87%E6%8A%A5%E7%A8%BF_images/image-20260420193920214.png)

这一阶段可以看到任务状态变成 `completed`，并且已经带上待复核列表，说明后端任务链路是闭合的。

### 步骤 6：结果汇总输出

最后使用报告导出器，把规则结果、解析信息和字段信息汇总成最终摘要。

```bash
python src\reporting\report_exporter.py --results-file outputs\s1_demo\04_rules.json --parse-file outputs\s1_demo\01_parse.json --fields-file outputs\s1_demo\02_fields.json --output outputs\s1_demo\06_summary.json
```
